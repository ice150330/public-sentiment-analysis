## 分析结果 [2026-07-08 10:10 CST]

**核心结论**：当前后端能力覆盖 UI.pen 需求的约 **30%**，存在约 **25 个待新增接口** 和 **17 个待建数据模型**。按 P0/P1/P2 分级推进，预计 3-4 个迭代周期可补齐。

## 四、推荐执行路线

### 第一阶段：P0 闭环（1-2 天）
目标：让现有 5 个页面从"有接口"变成"能跑通真实数据"

1. **扩展热榜接口**
   - `GET /topics` 返回 `sentiment` 联表数据（情感标签 + 置信度）
   - `GET /topics/{id}` 返回平台扩散信息、排名变化
   
2. **扩展统计接口**
   - `GET /stats/overview` 补充负面占比、昨日对比、下次采集时间
   - 统一所有列表接口返回 `{ items, pagination }` 格式
   
3. **扩展爬虫接口**
   - `GET /crawler/status` 补充真实队列长度、当前平台进度

### 第二阶段：P1 业务功能（3-5 天）
目标：支撑 UI.pen 中明显的一线业务功能

1. **预警模块**
   - `alert_rules` + `alert_events` + `alert_actions` 模型
   - `GET/POST/PUT /alert-rules` 规则管理
   - `GET /alerts` 预警队列
   - `POST /alerts/{id}/ack` `POST /alerts/{id}/resolve` 处置

2. **平台监测**
   - `GET /platforms/monitoring` 六平台状态矩阵
   - `GET /stats/platform-distribution` 平台分布
   - `GET /data-quality/freshness` 数据新鲜度

3. **数据质量**
   - `GET /data-quality/funnel` 处理漏斗
   - `GET /data-quality/checks` 质量检查项
   - `GET /data-quality/issues` 待处理问题

4. **系统管理**
   - `GET /system/health` 系统健康
   - `GET /system/logs` 系统日志
   - `GET /audit-logs` 审计日志

### 第三阶段：P2 高级分析（5-7 天）
目标：支撑聚类、预测、解释等高级能力

1. **聚类与传播**
   - `GET /topics/clusters` 聚类主题地图
   - `GET /topics/{id}/propagation` 传播路径
   - `GET /topics/{id}/related` 关联话题

2. **趋势预测**
   - `POST /forecast/heat` 热度预测
   - `GET /forecast/signals` 预测信号

3. **模型解释**
   - `GET /models/current` 模型状态
   - `POST /sentiment/explain` 词级贡献解释

## 五、关键风险点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 聚类/预测算法实现复杂 | P2 阶段延期 | 先用规则/统计方法占位，后续替换 ML |
| 微博/抖音反爬升级 | 爬虫失效 | 保留 mock fallback + 告警 |
| 数据库性能（热榜量大） | 查询慢 | 按日期分区 + 索引优化 |
| 异步任务队列 | 需要 Celery/Redis | 先用 SQLite + 线程池占位 |

## 七、验收标准

1. **每个 UI.pen 数据区域都有明确接口来源**，前端不再使用静态占位数据
2. **列表接口全部支持分页/筛选/排序**，统一返回 `{ items, pagination }`
3. **所有写操作有对应接口 + 审计记录**（启停/重试/保存/处置）
4. **统计接口返回 `last_updated`**，支撑顶栏同步状态
5. **新增接口补充到 `docs/API接口文档.md`**，含请求/响应示例
6. **每个新增接口至少 1 个正常用例 + 1 个错误用例测试**
