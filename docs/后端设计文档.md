### 1.1 选型理由

**SQLite 为何足够？**

| 对比维度 | SQLite | PostgreSQL |
|---------|--------|-----------|
| 安装配置 | 零配置，单文件 | 需安装服务，配置用户/权限 |
| 资源占用 | 内存 < 50MB | 内存 > 200MB |
| 毕设数据量 | 10万条记录轻松处理 | 大材小用 |
| 部署复杂度 | 复制文件即可 | 需安装/启动服务 |
| 论文写作 | 可写"轻量级嵌入式数据库选型" | 常规方案 |
| ACID | 完全支持 | 完全支持 |

**SQLite 在毕设场景下的限制与应对：**

| 限制 | 影响 | 应对方案 |
|------|------|---------|
| 无并发写入 | 爬虫+API同时写 | 使用队列批量写入，或单进程调度 |
| 无JSONB索引 | 原始文本存储查询慢 | 关键字段提取为独立列 |
| 无全文检索 | 文本搜索慢 | 使用 `fts5` 扩展（SQLite内置） |
| 单文件大小限制 | 数据量大时文件膨胀 | 定期归档旧数据，控制保留周期 |

### 2.2 表结构设计

#### 2.2.1 platforms（平台配置表）

```sql
CREATE TABLE platforms (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    name            VARCHAR(32) NOT NULL UNIQUE,           -- 英文标识: weibo, douyin, etc.
    display_name    VARCHAR(64) NOT NULL,                -- 中文显示名: 微博
    base_url        VARCHAR(255),                         -- 平台首页URL
    crawl_config    JSON,                                  -- 爬虫配置(JSON): {headers, selectors, ...}
    is_active       BOOLEAN DEFAULT TRUE,                 -- 是否启用采集
    sort_order      INTEGER DEFAULT 0,                     -- 排序权重
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_platforms_active ON platforms(is_active);
```

**初始数据：**

| id | name | display_name | base_url | is_active |
|----|------|-------------|----------|-----------|
| 1 | weibo | 微博 | https://weibo.com | TRUE |
| 2 | douyin | 抖音 | https://www.douyin.com | TRUE |
| 3 | toutiao | 今日头条 | https://www.toutiao.com | TRUE |
| 4 | baidu | 百度 | https://www.baidu.com | TRUE |
| 5 | bilibili | B站 | https://www.bilibili.com | TRUE |
| 6 | zhihu | 知乎 | https://www.zhihu.com | TRUE |

#### 2.2.2 hot_topics（热榜数据表）

```sql
CREATE TABLE hot_topics (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    platform_id     INTEGER NOT NULL,
    topic_id        VARCHAR(128) NOT NULL,               -- 平台原始话题ID
    title           VARCHAR(512) NOT NULL,                 -- 话题标题
    url             VARCHAR(1024),                        -- 详情链接
    heat_score      INTEGER,                               -- 热度值（平台原始值）
    category        VARCHAR(64),                          -- 分类标签
    content_summary TEXT,                                  -- 正文摘要（前500字）
    raw_data        JSON,                                  -- 原始数据备份(JSON)
    crawl_time      TIMESTAMP NOT NULL,                    -- 采集时间
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (platform_id) REFERENCES platforms(id) ON DELETE CASCADE,
    UNIQUE(platform_id, topic_id, crawl_time)              -- 联合唯一：同一平台同一话题同一天只存一次
);

-- 索引
CREATE INDEX idx_hot_topics_platform ON hot_topics(platform_id);
CREATE INDEX idx_hot_topics_crawl_time ON hot_topics(crawl_time);
CREATE INDEX idx_hot_topics_heat ON hot_topics(heat_score DESC);
CREATE INDEX idx_hot_topics_category ON hot_topics(category);
```

**设计说明：**
- `topic_id` 使用平台原始ID，确保跨平台唯一性由联合键控制
- `raw_data` 存储原始JSON，方便后续回溯和调试
- `crawl_time` 精确到分钟，同一天多次采集取最新
- 联合唯一键 `(platform_id, topic_id, crawl_time)` 防止重复入库

---

#### 2.2.3 sentiment_results（情感分析结果表）

```sql
CREATE TABLE sentiment_results (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    topic_id        INTEGER NOT NULL,                     -- 关联 hot_topics.id
    sentiment_label VARCHAR(16) NOT NULL,                -- positive | negative | neutral
    confidence      REAL NOT NULL,                         -- 置信度 0.0 ~ 1.0
    positive_score  REAL DEFAULT 0,                       -- 正面分数
    negative_score  REAL DEFAULT 0,                       -- 负面分数
    neutral_score   REAL DEFAULT 0,                        -- 中性分数
    model_version   VARCHAR(32),                          -- 模型版本，如 "bert-sentiment-v1"
    analyzed_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (topic_id) REFERENCES hot_topics(id) ON DELETE CASCADE,
    UNIQUE(topic_id)                                      -- 每条话题只分析一次
);

-- 索引
CREATE INDEX idx_sentiment_label ON sentiment_results(sentiment_label);
CREATE INDEX idx_sentiment_analyzed ON sentiment_results(analyzed_at);
```

**设计说明：**
- 三分类存储：label + 三分分数，便于后续分析置信度
- `model_version` 记录模型版本，支持模型迭代后重跑历史数据
- 与 `hot_topics` 一对一关系，通过 `topic_id` 外键关联

---

#### 2.2.4 crawl_logs（采集日志表）

```sql
CREATE TABLE crawl_logs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    platform_id     INTEGER NOT NULL,
    status          VARCHAR(16) NOT NULL,                 -- success | failed | partial
    records_count   INTEGER DEFAULT 0,                    -- 采集记录数
    error_message   TEXT,                                  -- 错误详情
    started_at      TIMESTAMP NOT NULL,                  -- 开始时间
    completed_at    TIMESTAMP,                            -- 完成时间
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (platform_id) REFERENCES platforms(id) ON DELETE CASCADE
);

-- 索引
CREATE INDEX idx_crawl_logs_platform ON crawl_logs(platform_id);
CREATE INDEX idx_crawl_logs_status ON crawl_logs(status);
CREATE INDEX idx_crawl_logs_time ON crawl_logs(started_at DESC);
```

**设计说明：**
- 记录每次采集的执行状态，便于监控和调试
- `status` 三种状态：success(全部成功)、failed(全部失败)、partial(部分成功)
- 可用于计算采集成功率、统计各平台数据量

### 2.3 数据库关系总结

| 关系 | 类型 | 说明 |
|------|------|------|
| platforms → hot_topics | 1:N | 一个平台有多条热榜记录 |
| hot_topics → sentiment_results | 1:1 | 一条热榜对应一条分析结果 |
| platforms → crawl_logs | 1:N | 一个平台有多条采集日志 |

### 2.4 核心查询示例

```sql
-- 查询今日各平台热榜数量
SELECT 
    p.display_name,
    COUNT(*) as topic_count
FROM hot_topics ht
JOIN platforms p ON ht.platform_id = p.id
WHERE date(ht.crawl_time) = date('now')
GROUP BY p.display_name;

-- 查询今日情感分布
SELECT 
    sentiment_label,
    COUNT(*) as count,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 2) as percentage
FROM sentiment_results
WHERE date(analyzed_at) = date('now')
GROUP BY sentiment_label;

-- 查询最近7天热度趋势（按平台）
SELECT 
    p.display_name,
    date(ht.crawl_time) as date,
    AVG(ht.heat_score) as avg_heat
FROM hot_topics ht
JOIN platforms p ON ht.platform_id = p.id
WHERE ht.crawl_time >= datetime('now', '-7 days')
GROUP BY p.display_name, date(ht.crawl_time)
ORDER BY date, p.display_name;

-- 查询采集成功率（最近30天）
SELECT 
    status,
    COUNT(*) as count,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 2) as percentage
FROM crawl_logs
WHERE started_at >= datetime('now', '-30 days')
GROUP BY status;
```

## 5. SQLite 性能优化建议

### 5.1 写入优化

SQLite 默认每个 INSERT 都提交事务，批量写入时极慢：

```python
# ❌ 慢：每次 INSERT 都自动提交
for topic in topics:
    db.execute("INSERT INTO hot_topics ...")

# ✅ 快：显式事务包裹批量插入
with db.begin():
    for topic in topics:
        db.execute("INSERT INTO hot_topics ...")
```

### 5.2 并发控制

SQLite 文件级锁，并发写入会冲突：

```python
# 方案：使用队列 + 单线程写入线程
from queue import Queue
from threading import Thread

write_queue = Queue()

def db_writer():
    """单线程写入守护进程"""
    while True:
        batch = write_queue.get()
        with db.begin():
            for record in batch:
                db.execute("INSERT INTO hot_topics ...")

writer_thread = Thread(target=db_writer, daemon=True)
writer_thread.start()

# 爬虫采集后推入队列
write_queue.put(batch_records)
```

### 5.3 定期维护

```sql
-- 清理过期数据（保留30天）
DELETE FROM hot_topics WHERE crawl_time < datetime('now', '-30 days');
DELETE FROM crawl_logs WHERE started_at < datetime('now', '-30 days');

-- 释放磁盘空间（VACUUM 重建数据库文件）
VACUUM;

-- 更新统计信息
ANALYZE;
```

### 5.4 FTS5 全文搜索（可选扩展）

```sql
-- 创建全文搜索虚拟表
CREATE VIRTUAL TABLE hot_topics_fts USING fts5(
    title, content_summary,
    content='hot_topics',
    content_rowid='id'
);

-- 自动同步触发器
CREATE TRIGGER hot_topics_ai AFTER INSERT ON hot_topics BEGIN
    INSERT INTO hot_topics_fts(rowid, title, content_summary)
    VALUES (new.id, new.title, new.content_summary);
END;

-- 全文搜索
SELECT * FROM hot_topics_fts WHERE hot_topics_fts MATCH '人工智能';
```
